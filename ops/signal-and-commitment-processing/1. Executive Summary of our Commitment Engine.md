
## A. Shared cross-signal commitment engine

### Purpose

Turn raw communication signals into structured commitment objects without pretending certainty.

### Core principle

Every source feeds the same engine.
The engine decides:

* is there a commitment candidate?
* who likely owns it?
* what was promised or requested?
* is it new, updated, completed, canceled, or unclear?
* should it be surfaced, quietly observed, or held for clarification?

### Engine stages

1. Ingest signal

   * accept normalized message/event from email, Slack, meetings, later others

2. Extract candidate units

   * split one signal into one or more possible commitment units

3. Classify speech act

   * request
   * self-commitment
   * acceptance
   * status update
   * completion/delivery
   * cancellation/decline
   * reassignment
   * informational only

4. Resolve fields

   * owner
   * requester
   * target/beneficiary
   * deliverable
   * due date/timing
   * evidence
   * blockers/dependencies

5. Link to lifecycle

   * create new
   * update existing
   * mark in progress
   * mark delivered
   * mark completed
   * cancel
   * reopen
   * ignore / observe only

6. Score confidence

   * commitment confidence
   * owner confidence
   * due-date confidence
   * completion confidence
   * merge/link confidence

7. Route outcome

   * surface now
   * quietly observe
   * hold for clarification
   * do not surface

### Shared final outcomes

* No commitment
* Observe only
* New commitment
* Multiple commitments
* Update existing commitment
* Reassign ownership
* Update date/scope/conditions
* Mark in progress
* Mark delivered
* Mark completed
* Cancel / decline
* Reopen
* Needs clarification
* Unresolved, do not surface

### Design rule

LLMs interpret language.
Deterministic logic controls:

* lifecycle transitions
* allowed outcomes
* confidence thresholds
* surfacing rules
* merge/update/create decisions

That is the right split for trust and consistency.

## B. Email-specific resolution map

### Purpose

Adapt the shared engine to email’s structure and ambiguity patterns.

### Email-specific preprocessing

Before any interpretation:

1. identify direction: sent vs received
2. identify sender, To, CC, named people
3. isolate only the latest authored message
4. remove quoted history from the current authored block
5. keep prior thread context available only for linking or clarification

This follows the strongest part of your current map. 

### Email-specific interpretation rules

1. Single-recipient email

   * direct asks usually target the recipient
   * self-commitments usually belong to the sender
   * confirmations usually resolve against the immediately prior ask

2. Multi-recipient email

   * do not assume ownership from recipient list alone
   * first try explicit names
   * then section-level addressing
   * then grammatical actor
   * then prior-thread linkage
   * otherwise keep ownership unresolved

3. Quoted history

   * do not detect new commitments from quoted text alone
   * use prior message context only to:

     * resolve confirmation
     * link updates
     * detect completion of an existing item
     * understand reassignment/cancellation

4. One email may create multiple outcomes

   * one message can contain:
[22 Mar 2026 09:29:47] Kevin Beeftink: * a new ask
     * a promise
     * a revised due date
     * a completion signal
   * process each candidate separately

### Executive email decision flow

* preprocess current email
* extract candidate units
* determine whether each unit is:

  * ask
  * promise
  * acceptance
  * update
  * completion
  * cancellation
  * reassignment
  * non-commitment
* resolve owner
* link to prior commitment if applicable
* score confidence
* choose outcome:

  * create
  * update
  * complete
  * cancel
  * reassign
  * clarify
  * observe
  * ignore
