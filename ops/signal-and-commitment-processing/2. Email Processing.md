## A. Shared cross-signal commitment engine — build-ready

## 1. Canonical input contract

Each source adapter must produce a normalized signal:

NormalizedSignal
- signal_id
- source_type                  // email, slack, meeting, etc.
- source_thread_id             // nullable for non-threaded inputs
- source_message_id
- occurred_at
- authored_at
- actor_participants[]         // who authored/spoke
- addressed_participants[]     // direct recipients / audience
- visible_participants[]       // all participants visible in context
- latest_authored_text
- prior_context_text           // optional, source-specific, bounded
- attachments[]
- links[]
- metadata{}

## 2. Candidate extraction contract

CommitmentCandidate
- candidate_id
- signal_id
- text_span
- normalized_text
- speech_act_candidate         // unknown until classified
- explicitness                 // explicit | implicit | mixed
- referenced_people[]
- referenced_dates[]
- referenced_deliverables[]
- evidence_markers[]

Rules:

* one signal may produce zero, one, or many candidates
* candidates must be span-linked to source text
* quoted/prior text may support linking but not candidate creation by default

## 3. Shared speech-act taxonomy

Allowed values:

* request
* self_commitment
* acceptance
* status_update
* completion
* cancellation
* decline
* reassignment
* informational

No other top-level class allowed in MVP.

## 4. Shared object contract

CommitmentObject
- commitment_id
- title
- normalized_deliverable
- owner_party_id              // nullable
- requester_party_id          // nullable
- beneficiary_party_id        // nullable
- due_at                      // nullable
- due_precision               // exact | day | week | vague | none
- source_of_truth             // signal / inferred / linked
- state                       // proposed | active | in_progress | delivered | completed | canceled | closed
- priority_score              // separate from confidence
- confidence_overall
- confidence_owner
- confidence_due
- confidence_completion
- clarification_needed        // bool
- clarification_reason[]      // owner, deliverable, due, target, existence
- linked_signal_ids[]
- linked_evidence_ids[]

## 5. Shared outcome contract

ResolutionOutcome
- action_type                 // ignore | observe | create | update | merge | split | complete | cancel | reopen | clarify
- resulting_commitment_ids[]
- rationale_codes[]
- confidence_band             // high | medium | low
- surface_recommendation      // now | digest | hold | suppress

## 6. Shared confidence bands

### High

Use when:

* strong speech act
* clear owner
* clear deliverable
* strong thread linkage if update/completion
* minimal contradictions

Allowed system behavior:

* can create or update automatically
* can surface in shortlist/main view
* can mark delivered if evidence is strong

### Medium

Use when:

* commitment likely exists
* one key field is inferred rather than explicit
* some ambiguity remains, but bounded

Allowed behavior:

* create as tentative or observe-first
* surface carefully worded suggestion
* do not auto-complete without stronger evidence

### Low

Use when:

* existence unclear
* owner ambiguous
* target ambiguous
* prior context required but unresolved
* multiple competing interpretations

Allowed behavior:

* observe only, or
* send to clarification queue, or
* suppress entirely

## 8. Shared lifecycle rules

### Create new commitment when:

* speech act is request, self_commitment, or acceptance
* no confident existing commitment link exists

### Update existing when:

* candidate references prior active commitment
* new content changes state, timing, scope, owner, or evidence

### Complete when:

* strong completion signal
* and strong link to an active commitment

### Cancel when:

* explicit decline, cancellation, or “no longer needed”
* linked to active commitment

### Reopen when:

* prior completed/canceled item is revived by new ask, follow-up, or failure signal

## 9. Shared hard rules

* Never let the LLM create a lifecycle state outside the enum
* Never let completion happen from weak lexical evidence alone if no commitment link exists
* Never treat quoted history as a new current commitment by default
* Never merge commitments without a merge-confidence gate
* Confidence and priority must remain separate

## B. Email-specific resolution map — build-ready

## 1. Email adapter input additions

EmailSignal extends NormalizedSignal
- direction                    // sent | received
- sender
- to[]
- cc[]
- bcc_present                  // bool, no identities required in MVP
- subject
- message_id
- in_reply_to_message_id
- references[]
- latest_authored_block
- quoted_history_blocks[]

## 2. Email preprocessing nodes

### Node E1 — Direction detection

Decision:

* is this email sent by the user or received by the user?

Outputs:

* direction = sent
* direction = received

### Node E2 — Participant extraction

Extract:

* sender
* To recipients
* CC recipients
* named individuals in current authored block

### Node E3 — Latest authored block isolation

Rule:

* only the newest authored block is eligible for new candidate extraction

Quoted history may be used only for:

* confirmation linking
* update linking
* completion linking
* reassignment/cancellation context

This is directly aligned with your current draft. 

### Node E4 — Candidate segmentation

Split latest authored block into candidate units by:

* sentence boundaries
* bullets/numbered items
* greeting/body/signoff exclusion
* conjunction-based splitting when multiple deliverables are present

Output:

* zero to N email candidates

## 3. Email decision tree per candidate

## Node E5 — Is this commitment-relevant?

Classify candidate as one of:

* request
* self_commitment
* acceptance
* status_update
* completion
* cancellation
* decline
* reassignment
* informational

Fallback:

* if classifier confidence is low and no strong action phrase exists → informational / observe only

---

## 4. Owner resolution tree

## Node E6 — Explicit named owner?

If candidate explicitly names owner:

* owner = named person
* confidence_owner = high

Examples:

* “Sarah will send it”
* “John, can you handle this?”

## Node E7 — Grammatical actor resolution

If owner not explicit:

* for self_commitment → owner = sender
* for direct ask in single-recipient email → owner = recipient
* for completion statement in sent email → owner = sender
* for status update from sender about their work → owner = sender

## Node E8 — Addressed-subsection resolution

For multi-recipient emails:

* detect whether section is clearly addressed to one recipient
* if yes, owner = addressed person
* if no, continue

This is the missing formalization your current map needs. The draft correctly senses this, but it needs a stricter branch. 

## Node E9 — Prior-thread linkage

If current message is:

* confirmation
* acceptance
* update
* completion
* cancellation

Then check prior message for linked ask/commitment.

Bounded rule:

* use immediate previous authored message first
* only expand further back if previous message points backward ambiguously

## 5. Deliverable resolution tree

## Node E11 — Extract deliverable

Resolve the promised/requested object or action.

Sources:

* explicit action phrase
* referenced noun phrase
* attachment/link mention
* prior linked ask if current message is acceptance/completion

Fallback:

* if deliverable remains vague (“this”, “that”, “it”) and no strong prior link exists:

  * clarification_needed = true
  * confidence_overall capped at medium

## 6. Due date resolution tree

## Node E12 — Extract timing

Resolve:

* exact date/time
* day-level
* week-level
* vague timing
* none

Fallback:

* if timing vague (“soon”, “later”, “ASAP”):

  * keep due_at null or low precision
  * due_precision = vague
  * do not block commitment creation if owner + deliverable are strong

## 7. Lifecycle linkage tree

## Node E13 — Is this new or linked?

Try to match candidate against existing commitments by:

* thread
* participants
* deliverable similarity
* recency
* state compatibility

Outcomes:

* no confident match → create new
* one strong match → update linked
* multiple plausible matches → clarification or observe only

## Node E14 — Allowed lifecycle transitions

By speech act:

### request

* create new
* or update existing unmet request

### self_commitment

* create new
* or update existing if clearly same deliverable

### acceptance

* update existing request with accepted owner
* or create if prior request not already represented

### status_update

* update to in_progress or blocked
* never create new by default unless explicit new commitment appears too

### completion

* mark delivered or completed
* only if linked strongly enough

### cancellation / decline

* cancel or mark not proceeding
* only if linked strongly enough

### reassignment

* update owner
* preserve history of previous owner

## 8. Completion rules for email

## Node E15 — Completion evidence strength

### Strong completion evidence

* explicit “done”, “sent”, “attached”, “here it is”
* and attachment/link present, or
* clear reply to linked request, or
* explicit mention of delivered artifact

Outcome:

* mark delivered
* mark completed only if commitment type allows completion on send

### Medium completion evidence

* “handled”
* “should be there”
* “I took care of it”
* without attachment or clear artifact

Outcome:

* mark delivered only if linked strongly
* otherwise observe update only

### Weak completion evidence

* “thanks”
* “great”
* “noted”
* “looks good”

Outcome:

* do not complete

Hard rule:

* lexical completion phrase without commitment linkage cannot close a commitment

## 9. Clarification routing rules

## Node E16 — When to clarify vs observe

### Clarify when:

* commitment likely exists
* and user-facing value is high
* and one blocking field is missing:

  * owner
  * deliverable
  * due date for externally relevant item
  * target/beneficiary

### Observe only when:

* commitment existence is medium/low
* ambiguity may resolve naturally in the next reply
* external interruption would add friction

### Suppress when:

* candidate is too weak
* or mostly informational
* or contradicted by stronger evidence

This matches the development brief’s emphasis on clarification, silent observation windows, and confidence-based routing.

## 10. Email confidence rules

Use a point model or weighted dimensions, but keep these dimensions explicit.

Increase for:

* imperative phrasing
* commitment verbs
* specific deliverable
* clear recipient or actor

Decrease for:

* hypothetical language
* brainstorming
* courtesy language only
* quoted-text-only evidence
* conflicting interpretations

## 11. Email-specific hard guardrails

* Do not create commitments from signatures, greetings, disclaimers
* Do not create new commitments from quoted history
* Do not infer owner purely from CC
* Do not mark completed from polite acknowledgement alone
* Do not merge two deliverables in one sentence unless same object/action is clear
* Do not let org-role heuristics override explicit textual evidence in MVP
* Do not search whole-thread history unless immediate prior context fails

That last point matters because your draft already leans toward “use more thread context when unclear,” but I would keep that bounded, not default-expansive.

## 12. Recommended MVP surfacing behavior for email

### Surface now

* high-confidence new commitment
* high-confidence due-date change
* high-confidence completion of an important active item
* reassignment affecting responsibility

### Quietly observe

* medium-confidence ask/promise with missing due date
* status updates without strong action need
* likely completion with weak evidence

### Clarification queue

* likely commitment, unclear owner
* likely commitment, unclear deliverable
* multiple plausible matches to prior commitment

### Suppress

* informational-only
* quoted-history-only
* weak, non-actionable ambiguity
