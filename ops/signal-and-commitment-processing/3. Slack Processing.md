# Slack-Specific Resolution Map (v1)

## 1. Locked MVP assumptions

For Slack in MVP, the map assumes:

* Slack is **internal only**
* Threads are strong context, but **non-threaded messages still get processed immediately**
* Later replies can enrich earlier signals
* Non-threaded linkage is based on **topic cluster**, not just time window
* Edits are **new signal events on the same source message**, not full source reinterpretation
* Reactions are ingested as **auxiliary evidence**, not primary truth
* Small practical commitments are valid candidates, but not all should be surfaced
* Standalone completion phrases like “done” are weak unless linked to a commitment
* Slack should generally be stricter than email on vague ownership because the channel is noisier and more fragmented

That structure is consistent with the broader brief’s emphasis on channel-specific caveats, edits, threading, completion evidence, silent observation windows, confidence separation, and trust-over-cleverness product behavior. 

---

## 2. Slack source types

Every Slack signal starts by classifying the source shape:

1. **DM**
2. **Private channel**
3. **Public channel**
4. **Thread parent**
5. **Thread reply**
6. **Edited message event**
7. **Reaction event**
8. **Attachment / link-bearing message**

This matters because Slack meaning depends heavily on message placement and reply structure, much more than email.

---

## 3. Entry flow

### Step 1 — Ingest raw Slack event

Normalize:

* message ID
* channel / DM ID
* sender
* mentions
* thread_ts / parent reference
* timestamp
* edit timestamp / version
* reactions
* attachments / links / files
* surrounding nearby messages for fallback context

### Step 2 — Determine message context class

Branch into one of four context modes:

#### A. Thread reply

Use:

* parent message
* earlier replies in thread
* reply ordering

This is the strongest Slack context.

#### B. Thread parent

Treat as standalone signal first.
Later thread replies may enrich or modify it.

#### C. Non-threaded channel message

Treat as standalone signal immediately.
Also open a **topic-cluster candidate** for possible follow-up linkage.

#### D. Edited message

Treat edit as a new signal event attached to the same message source.
Compare prior and edited text for:

* added owner info
* added timing
* added deliverable detail
* changed intent
* changed completion/progress meaning

This matches your instinct while keeping the data model cleaner than pretending it is a new independent message each time. The brief’s versioning, audit history, linked-signal model, and replay/debugging thinking all point in that direction. 

---

## 4. Candidate detection

For each Slack message, extract zero, one, or multiple candidate units.

A single Slack message may contain:

* a request
* an acceptance
* a promise
* a status update
* a completion signal
* a reassignment
* a cancellation
* pure information

If no candidate unit exists:

* outcome = **Ignore / informational only**

If one or more candidate units exist:

* process each separately

---

## 5. Speech-act classification

Each candidate unit gets classified into one of these primary types:

### A. Direct request

Examples:

* `@Joe can you update this?`
* `Can someone send the file?`
* `Need this done before 3`
* `Who can handle the client reply?`

Default meaning:

* speaker is requester
* some other party is being asked to act

### B. Self-commitment

Examples:

* `I’ll handle it`
* `I can take this`
* `I’ll send it after lunch`
* `Let me look into it`

Default meaning:

* speaker is likely owner

### C. Acceptance / pick-up

Examples:

* `Yep, I’ll do it`
* `On it`
* `Got it`
* `I can take that`

Default meaning:

* speaker is accepting an ask or implicit task from prior context

### D. Status / progress update

Examples:

* `Working on it`
* `Waiting on design`
* `Almost done`
* `Blocked on legal`

Default meaning:

* modifies an existing commitment if linked

### E. Completion / delivery signal

Examples:

* `Done`
* `Sent`
* `Handled`
* `Pushed`
* `Here it is`
* attachment / link reply

Default meaning:

* possible delivery/completion evidence, but not always full closure

### F. Reassignment / delegation

Examples:

* `@Sarah will take this`
* `Looping in Tom for this`
* `Can Dino handle ads side?`

Default meaning:

* ownership may move or become newly assigned

### G. Cancellation / decline / no longer needed

Examples:

* `Never mind`
* `No need anymore`
* `I can’t take this`
* `Let’s drop it`

Default meaning:

* active or candidate commitment may be cancelled / declined / closed

### H. Informational only

Examples:

* `FYI this changed`
* `The client replied`
* `Looks like prod is down`

Default meaning:

* may still become context for later commitments, but not a commitment itself

This aligns well with the brief’s framing around explicit/implicit commitments, source signals vs commitment objects, candidate creation rules, clarification, and completion evidence.

## 6. Owner resolution map

This is where Slack needs its own source-specific logic.

### Highest-confidence owner resolution

Use in this order:

1. **Explicitly named / mentioned owner**

   * `@Joe can you fix this?`
   * owner = Joe

2. **Direct grammatical actor**

   * `I’ll handle it`
   * owner = speaker

3. **Linked acceptance**

   * parent: `Can someone take this?`
   * reply: `I got it`
   * owner = replier

4. **Clear delegated owner**

   * `Sarah will send it`
   * owner = Sarah

### Medium-confidence owner resolution

Use only when context is strong:

5. **Topic-cluster pickup**

   * message 1: `Need updated numbers before 2`
   * message 2: `I’ll do it`
   * owner = second speaker if same topic cluster is strong

6. **Section / turn-taking clue**

   * especially in small group channel exchanges where the target is obvious from adjacent turns

### Low-confidence / unresolved

Do not auto-resolve:

7. `Can someone do this?`
8. `We should fix this`
9. `Someone needs to reply`
10. `Can you look?` when multiple possible addressees exist and no mention is present

Outcome:

* **Needs clarification**
* or **observe only**
* or **store unresolved candidate**

### Slack-specific rule

Be stricter than email on vague ownership.
Slack is noisier, shorter, and more fragmented, so false certainty is more dangerous.

## 7. Non-threaded topic-cluster logic

This is the big Slack-specific addition.

When a message is not in a thread, Rippled should still process it immediately. But later nearby messages may belong to the same conversational unit even if nobody used threads.

So create a **topic-cluster layer**.

A later non-threaded message can be linked to an earlier one when several of these align:

* same channel
* reasonably close in time
* overlapping participants / mentions
* overlapping named entities or artifacts

  * file
  * bug
  * ticket
  * customer
  * page
  * campaign
* semantic similarity
* adjacency / turn-taking pattern
* no stronger competing topic between them

### Example: one cluster

1. `@Joe can you update the pricing block before the client call?`
2. `Yep`
3. `Need it in 20`
4. `On it`
5. `Done`

Even without a thread, that is one likely cluster.

### Example: competing topics

1. `@Joe can you update the pricing block before the client call?`
2. `Yep`
3. `Who is handling the invoice issue?`
4. `I’ll take invoice`
5. `Done`

Now `Done` is ambiguous unless linked carefully.

### Context priority order

Use Slack context in this order:

1. thread context
2. topic cluster
3. immediate adjacency
4. otherwise unresolved

This fits the normalization, linked-signal, merge, and history-preservation logic already implied in your brief.

## 8. Field extraction

For each candidate commitment, resolve:

* owner
* requester
* deliverable / action
* beneficiary / target
* timing / due date
* channel context
* evidence attached
* blockers / conditions
* confidence

### Field rules

If owner missing:

* do not invent unless very strongly implied

If action vague:

* keep candidate but lower confidence

If timing vague:

* active commitment may still exist

If deliverable unclear:

* keep as candidate, likely observe first

If conflicting context:

* unresolved / clarification path

## 9. Lifecycle linking

Before creating a new commitment, check if the Slack signal is actually:

1. **new commitment**
2. **acceptance of existing request**
3. **update to existing commitment**
4. **completion evidence**
5. **reassignment**
6. **cancellation / decline**
7. **reopen signal**
8. **duplicate restatement**

### Create new commitment when

* request + likely owner
* self-commitment
* acceptance clearly picking up prior ask
* direct delegation with clear new owner

### Update existing commitment when

* progress signal
* timing change
* scope change
* blocker appears
* clarification added by edit or reply

### Close / deliver when

* strong completion evidence linked to active commitment

### Reopen when

* new follow-up indicates earlier completion was partial, failed, or needs rework

This is exactly why lifecycle, merge, and completion should stay distinct layers in the spec stack. 

---

## 10. Edit handling map

Edits should be processed as **versioned signal events** on the same Slack message.

### If edit only changes spelling / formatting

* store edit event
* no commitment change

### If edit adds commitment meaning

Example:

* original: `I’ll send`
* edited: `I’ll send by 4`

Outcome:

* enrich timing field
* raise confidence

### If edit changes scope or owner

Example:

* original: `I’ll handle this`
* edited: `Sarah will handle this`

Outcome:

* reassessment of owner
* possible reassignment / linked correction
* preserve prior state in history

### If edit weakens commitment

Example:

* original: `I’ll send it today`
* edited: `I’ll try to send it today`

Outcome:

* lower confidence
* maybe shift from strong commitment to softer candidate

So yes: your approach makes sense, but I’d formalize it as a versioned update-signal, not a totally separate free-floating message.

---

## 11. Reaction handling map

Reactions should be ingested now, but treated as weak evidence in MVP.

### Suggested interpretation tiers

#### Seen / triaged style

* 👀
* generally means noticed or reviewing
* not ownership
* not completion

#### Acknowledgement / acceptance-adjacent

* 👍
* 👌
* may mean agreement or acknowledgement
* weak acceptance evidence only

#### Positive / approval style

* ✅
* ☑️
* 💯
* may support approval or closure, but too ambiguous to stand alone

#### Team-custom semantics

* 🚀
* 🧵
* 🔥
* can be meaningful in specific teams, but too unstable for MVP truth

### MVP reaction rule

* reactions can **slightly influence confidence**
* reactions can **enrich evidence trail**
* reactions cannot independently:

  * create a commitment
  * assign ownership
  * mark full completion
  * prove deployment
  * prove external delivery

Your instinct to still digest them as signals is right. They belong in the evidence layer now, even if their operational meaning stays weak for MVP. That fits the product principle of capturing more than you show, and inferring more than you assert. 

---

## 12. Completion evidence ladder

Slack needs a strong evidence ladder because “done” often means very different things.

### Strong completion

* direct reply to request + artifact
* direct reply + attachment
* direct reply + document link
* direct reply + PR / deploy / commit / bot evidence
* explicit delivery phrase tied to known commitment

Examples:

* `Sent` + PDF
* `Done, here’s the deck`
* `Pushed` + PR link
* `Updated: [doc link]`

Outcome:

* **delivered** or **completed** with high confidence

### Medium completion

* thread-linked `done`
* thread-linked `handled`
* thread-linked `sent`

Outcome:

* likely delivered/completed
* maybe not fully closed yet

### Weak completion

* non-threaded `done`
* `sent it`
* `handled`
  with no strong topic linkage

Outcome:

* do not auto-close
* maybe update confidence or mark possible completion evidence

### Important rule

Slack completion language does not automatically prove:

* deployed to production
* sent to external stakeholder
* approved by recipient
* fully closed end-to-end

It may only prove:

* speaker finished their part
* file was sent
* draft was prepared
* internal handoff occurred

That is fully in line with the brief’s emphasis on commitment-type-specific completion rules and evidence models. 

---

## 13. Clarification / wait logic

Slack should often wait before surfacing ambiguous things.

### Clarify or hold when:

* owner unresolved
* action vague
* multiple possible commitments in same cluster
* completion phrase unlinked
* topic cluster weak
* reaction-only evidence
* collective language like `we should` with no pickup

### Observe silently when:

* small practical commitment exists but is weak
* likely commitment but missing timing or target
* acceptance is probable but not certain
* completion is plausible but not strong enough

### Surface sooner when:

* clear owner
* clear action
* clear timing
* explicit ask or promise
* artifact/evidence included
* repeated follow-up indicates importance

This lines up with the brief’s silent observation windows, confidence thresholds, and surfacing-vs-detection separation. 

---

## 14. Surfacing rules for Slack

### Surface immediately when

* direct ask to a named person
* explicit self-commitment with action and timing
* clear pick-up of request
* strong reassignment
* strong completion evidence on already surfaced commitment

### Hold internally first when

* `I’ll check`
* `let me look`
* `I’ll ask them`
* `on it`
* weak `done`
* reaction-only acknowledgement
* unresolved channel ask

### Escalate to surfaced later if

* no follow-up occurs within observation window
* later reply adds timing
* later reply confirms ownership
* the same topic appears again
* the user has a preference for surfacing small commitments

This is where Slack can be generous in capture but selective in user burden.

---

## 15. Final Slack outcome set

For each Slack signal or linked cluster, Rippled should end with one of these bounded outcomes:

1. **Ignore / informational only**
2. **Observe only**
3. **Create new commitment**
4. **Create multiple commitments**
5. **Update existing commitment**
6. **Assign or resolve owner**
7. **Reassign owner**
8. **Update due date / scope / condition**
9. **Mark in progress**
10. **Mark blocked / waiting**
11. **Mark delivered**
12. **Mark completed**
13. **Cancel / decline / not proceeding**
14. **Reopen commitment**
15. **Needs clarification**
16. **Unresolved, do not surface yet**
17. **Store auxiliary evidence only**
    (best for reactions and weak completion indicators)

That is the bounded complete outcome map for Slack.
