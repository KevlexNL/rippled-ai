# Phase 05 — Completion Detection: Decisions

## Trinity Pre-Build Approvals (from interpretation.md review)

### Q1 — New completion_signals table vs JSONB vs existing table?
**Decision: Use existing `CommitmentSignal` table with `signal_role=delivery`.**

Rationale: `CommitmentSignal` already has `confidence`, `interpretation_note`, indexed on `commitment_id` + `source_item_id`. A new table would duplicate structure. JSONB would lose queryability for Phase 06 surfacing. The existing table is the right fit.

### Q2 — Migration for `delivered_at` + `auto_close_after_hours`?
**Decision: YES — add both columns via Alembic migration.**

- `delivered_at`: needed because `state_changed_at` is overwritten on every transition; auto-close sweep and surfacing both need delivery timestamp
- `auto_close_after_hours`: nullable INT with server_default=48; allows per-commitment configuration in future; better to add now than migrate a live table later

### Q3 — Pydantic vs plain dict vs dataclass for scorer output?
**Decision: `@dataclass(frozen=True)` — consistent with TriggerPattern in patterns.py. No Pydantic overhead for internal-only objects.**

### Q4 — Threshold values?
**Decision: active→delivered: delivery_confidence >= 0.65 AND evidence_strength != "weak"**
**Decision: delivered→closed: closure_readiness_confidence >= 0.75 AND age >= auto_close_after_hours**

Reasoning: 0.65 maps to "actor sent something credible but not fully verified". 0.75 for closure is a higher bar — composite score must be strong across multiple dimensions.
Log-only zone: 0.40–0.65 — write signal, no state change.

### Q5 — Auto-close sweep in Phase 05 or Phase 06?
**Decision: Phase 05 owns auto-close sweep.**

Closure is a lifecycle decision, not a surfacing decision. Phase 06 should receive commitments already in correct state.

## Implementation Decisions

### D1 — Service structure
```
app/services/completion/
├── __init__.py
├── matcher.py
├── scorer.py
├── updater.py
└── detector.py
```

### D2 — Actor match is required (hard gate)
A source_item with no actor match against commitment.resolved_owner or commitment.suggested_owner returns 0 matches. Prevents false completions from unrelated activity.

### D3 — observe_until guard in matcher
Commitments with `observe_until > now()` are excluded from matching (still in observation window). This edge case preserves Phase 04's observation window semantics.

### D4 — 30-minute lookback window in sweep (not 10-minute)
Beat fires every 10 minutes but queries source_items ingested in last 30 minutes. Overlap is intentional to handle beat jitter and processing delays. Idempotency via CommitmentSignal UniqueConstraint prevents duplicate writes.

### D5 — `signal_role` value for delivery signals
Use string `"delivery"` — consistent with how Phase 04 used signal_role strings. Not introducing a new enum.

### D6 — Commitment types for scoring penalty
- `send`, `share`, `deliver`, `introduce` → ×0.95 completion multiplier
- `follow_up`, `update`, `coordinate` → ×0.80
- `review`, `check`, `investigate`, `look_into` → ×0.70
- All others → ×0.75

### D7 — `delivered_at` is set in updater on active→delivered transition
The `updater.apply_completion_result()` sets `commitment.delivered_at = now()` when transitioning from `active`. Not overwritten on subsequent signals.

### D8 — Celery task name: `run_completion_sweep`
Adds to the existing beat schedule in `app/tasks.py`. Interval: 10 minutes.

## Build-time Implementation Decisions (Phase 05 BUILD stage)

### D9 — Delivery keyword alone does not qualify a match
`find_matching_commitments` requires at least one of (deliverable keyword overlap, thread continuity, recipient match). A delivery keyword ("sent", "shared", etc.) is tracked in `matched_patterns` as supporting evidence but does not gate the match.

**Why:** Without this constraint, every outbound "sent" email matches ALL active commitments for that actor. The matcher must identify WHICH commitment was delivered, not just that something was delivered.

**Implication:** `evidence_strength="weak"` is never generated by the production matcher path. Weak evidence is only constructable directly for scorer unit tests.

### D10 — Strong evidence requires deliverable + (thread OR outbound+attachment+recipient)
`_compute_evidence_strength` classifies strong as: actor + deliverable + (thread OR (outbound + attachment + recipient match)). This reconciles §3a channel classification ("outbound with attachment to known recipient = strong") with §3b rule ("deliverable + thread = strong").

### D11 — Python-side time check in auto-close sweep
Auto-close sweep queries delivered commitments, then applies `now >= delivered_at + timedelta(hours=auto_close_after_hours)` in Python. Avoids SQLAlchemy dialect-specific interval arithmetic while keeping test mocks simple.

### D12 — `_load_origin_thread_ids` pre-loads thread context before matcher
Detector pre-loads origin CommitmentSignal → SourceItem.thread_id and attaches as `commitment._origin_thread_ids` so the matcher stays a pure function (no DB parameter).

### D13 — Test factory defaults set target_entity=None / recipients=[] when testing non-recipient behavior
Default `make_source_item` uses `recipients=["bob@example.com"]` and `make_commitment` uses `target_entity="Bob"`. Tests focused on quoted suppression or deliverable matching must explicitly remove these to avoid unintended recipient match firing. Documented in test file comments.
